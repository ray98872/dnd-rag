i

Sparse vector collision fix

Hashing tokens to a 100,000-bucket index space causes collisions. The fix: accumulate TF values into a dict keyed by index before converting to lists — so two tokens that hash to the same bucket merge their weights rather than producing a Qdrant validation error for duplicate indices.

03

Hybrid retrieval & reranking

Two-stage retrieval

  • Dense search — BGE cosine similarity over 384-d vectors. Strong on semantic meaning: finds chunks that are conceptually similar to the query even without keyword overlap.
  • Sparse search — BM25-style TF index. Strong on exact terminology: critical for rules queries like "Grappled condition" where the precise word matters.
  • RRF fusion — Reciprocal Rank Fusion combines the two ranked lists into a single top-10 shortlist, avoiding the need to tune a score-blending weight.

ii

Why hybrid retrieval?

Pure semantic search misses rules that share vocabulary with unrelated concepts. Pure keyword search misses paraphrased questions. Hybrid captures both — particularly important for a rules corpus where exact terminology is load-bearing.

iii

qdrant-client 1.18 breaking change

The client.search() method was removed in qdrant-client 1.18. Both dense and sparse search now use client.query_points() with a using= parameter to select the named vector — the response is accessed via response.points.

RAGAS evaluation

10-sample evaluation generated synthetically from the corpus using Llama 3.3 70B, then scored with RAGAS 0.4.x using the same model. Embeddings from BAAI/bge-small-en-v1.5.

Faithfulness

0.79

Fraction of claims in the answer that are grounded in the retrieved context. The oracle rarely hallucinates.

Answer Relevancy

0.69

How directly the answer addresses the question. Score reflects over-verbose answers that contain the right information but add filler.

Context Recall

0.60

How much of the ground-truth answer is covered by the retrieved chunks. Lower score reflects some retrieval gaps on specific stat lookups.

iv

Synthetic dataset generation

30 question–answer pairs were generated by prompting the LLM with random corpus chunks and asking for a specific, answerable rules question plus a ground-truth answer. This avoids the need for manual annotation while remaining grounded in real SRD content.

v

RAGAS 0.4.x migration

RAGAS 0.4 replaced its evaluation API: samples are now SingleTurnSample objects collected into an EvaluationDataset; LLM and embeddings must be wrapped in LangchainLLMWrapper / LangchainEmbeddingsWrapper; and metrics return per-sample lists rather than a single float — requiring np.nanmean for aggregation.
